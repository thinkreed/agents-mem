**维护**: 产品决策时更新 | 状态: active

## 一、产品愿景

**面向多 Agent 系统的渐进式记忆层**

核心目标: **Token 成本节省 80-91%**

通过四层渐进式披露，Agent 只加载需要的信息层级，避免一次性加载全部内容。

---

## 二、目标用户

| 角色 | 需求 |
|------|------|
| Agent 开发者 | 为 Agent 提供高效记忆系统 |
| 多 Agent 编排者 | 169+ Agent 的隔离与共享 |
| 中文用户 | 原生中文语义搜索支持 |

---

## 三、核心价值

### 1. Token 效率

| 场景 | 传统方式 | 本系统 | 节省 |
|------|----------|--------|------|
| 快速浏览 | 加载全文 (10k+ tokens) | L0 (~100 tokens) | 99% |
| 上下文理解 | 加载全文 | L1 (~2k tokens) | 80% |
| 深度分析 | 加载全文 | L2 (全文) | 0% (按需) |

### 2. 多租户隔离

- 用户级隔离 (userId 必填)
- Agent 级隔离 (可选)
- 团队级共享 (可选)

### 3. 语义搜索

- 原生中文支持
- 混合搜索 (FTS + Vector)
- 语义理解 (非精确匹配)

---

## 四、设计原则

### 1. Scope 第一

所有操作必须提供 Scope，确保数据隔离。

### 2. 渐进式披露

从不一次性加载全部内容。从 L0 开始，按需深入。

### 3. 事实追溯

每个事实都可以追溯到原始来源，确保可信度。

### 4. 异步 Embedding

不阻塞用户操作。Embedding 后台处理。

---

## 五、用户旅程

### 创建文档

```
1. 用户调用 mem_create(document, scope, content)
2. 存储到 SQLite (同步)
3. 存储到 OpenViking (同步)
4. 排队 Embedding (异步)
5. 返回 { id, uri }
```

### 搜索文档

```
1. 用户调用 mem_read({ search: "关键词", scope })
2. OpenViking 混合搜索
3. 返回结果 + L0 摘要
4. 用户可选择加载 L1/L2
```

### 追溯事实

```
1. 用户调用 mem_read({ id: "fact-123", trace: true })
2. 获取事实
3. 追溯 source_id → tiered → original_uri
4. 返回完整追溯链
```

## 六、竞争差异

| 特性 | 本系统 | 传统向量数据库 |
|------|--------|---------------|
| 分层加载 | L0/L1/L2 三级 | 无 |
| 中文支持 | 原生语义搜索 | 需额外配置 |
| 事实追溯 | 内置追溯链 | 无 |
| 多租户 | Scope 原生支持 | 需自行实现 |
| Token 优化 | 核心设计目标 | 不考虑 |
